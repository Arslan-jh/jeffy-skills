# Architecture Map

> **方法论定位**
> 架构图不是信息图、组织结构图或功能清单。它是一页解释模型：把复杂对象压缩成“角色、输入、价值流、能力层、执行机制、支撑约束、战略结果”的可视化结构，让读者在 30 秒内回答：这是什么、怎么运作、为什么重要、受什么限制。

你正在执行一次全局架构图任务。默认交付物是：**一份可直接打开的一页式 HTML/SVG 架构图**。除非用户明确只要文本草图，否则不要只输出 Mermaid、列表或口头说明。

## 0. 先判断任务是否需要本 Skill

触发后先做 10 秒判断：用户需要的是“架构可视化”还是“普通解释”。

- **直接开始**：用户要求全局架构图、一图讲清楚、业务架构、系统架构、生态图、能力地图、价值链图、平台架构、运营模型，或要求“拆解 XX 的架构”。
- **最多追问一次**：读者、使用场景或对象边界会显著改变图的结构。一次只问最关键的 1-3 个问题。
- **降级处理**：用户只是问“XX 是什么”或只要短解释时，给简短回答，不强行生成正式图。
- **必须联网核验**：对象涉及当前公司、市场、产品、价格、监管、人物、财务、竞品、近期事件或“最新/现在/最近”状态时，先查公开来源，再把信息当事实使用。

若直接开始，记录以下任务设定：

| 字段 | 要求 |
|---|---|
| 架构对象 | 公司、产品、系统、行业、平台、AI/Agent、流程、组织能力或其他 |
| 读者视角 | 老板/投资人/产品/技术/运营/销售/用户/公众；未指定则面向业务决策者 |
| 使用场景 | 汇报、研究、战略讨论、产品规划、技术解释、方案评审；未指定则用于汇报 |
| 时间基准 | 使用当前日期；涉及当前状态必须联网确认 |
| 输出形式 | 默认 HTML/SVG 一页图；如用户指定 PPT/PDF/图片则按指定格式 |

## 1. 先找事实，再做解释

架构图可以包含解读，但不能把解读伪装成事实。正式绘图前先建立事实边界。

### 1.1 来源优先级

| 信息类型 | 优先来源 |
|---|---|
| 公司业务/财务 | 年报、季报、交易所公告、投资者关系材料、监管文件 |
| 产品/技术 | 官方文档、发布说明、产品页面、开发者文档、论文、标准 |
| 市场/竞争 | 财报、行业协会、权威媒体原创报道、分析师报告、公开数据 |
| 用户/运营 | 用户评价、社区讨论、案例、截图、用户提供材料 |
| 监管/风险 | 法规文本、监管公告、处罚文件、诉讼材料、合规说明 |

### 1.2 事实标签

图中或交付说明中使用四类标签：

| 标签 | 含义 | 用法 |
|---|---|---|
| 官方事实 | 来自官方公告、财报、文档、监管文件 | 可支撑关键结构判断 |
| 观察事实 | 来自用户材料、截图、可见产品界面 | 可支撑界面、流程、功能判断 |
| 分析解读 | 基于事实抽象出的层级、飞轮、控制点、能力模型 | 必须标注为解读，不代表官方术语 |
| 待确认 | 缺少可靠来源但会影响架构的假设 | 放在底部或侧栏，不进入核心事实 |

### 1.3 最低证据门槛

1. 事实节点至少有官方事实或可靠二手来源。
2. 因果/控制点判断至少有事实节点和行为证据支撑。
3. 战略命名、飞轮、平台层、操作系统、Agent OS 等强解释词，除非官方这样表述，否则标为分析解读。
4. 找不到可靠来源时，降低确定性，不要用“显然、必然、本质上”包装。
5. 公开信息不足时，仍可画图，但必须在图底部写清关键假设。

## 2. 架构建模协议

正式绘图前，必须先建立内容模型。不要边画边想。

每张图必须回答七个问题：

1. **核心判断**：这张图要证明或解释什么？
2. **关键角色**：谁参与系统，谁受益，谁承担成本？
3. **输入/触发**：需求、数据、资源、市场力量或外部事件从哪里来？
4. **处理机制**：系统如何完成匹配、理解、编排、交易、执行、履约或反馈？
5. **核心能力**：哪些能力是控制点、差异化来源或瓶颈？
6. **底层支撑**：哪些数据、模型、技术、供应链、组织、治理、资本或基础设施支撑系统？
7. **结果与约束**：最终产生什么价值，同时受什么风险、成本、合规、可靠性或竞争约束限制？

如果七个问题中有两个以上无法回答，先补充材料或标注假设，不要直接生成图。

### 2.1 内容模型模板

正式画图前，在内部形成如下模型：

```text
主题：
核心判断：
读者视角：
事实边界：

角色：
- ...

输入/触发：
- ...

主价值流：
触发 -> 接入 -> 匹配/理解 -> 编排/交易 -> 执行/履约 -> 反馈/复购

核心能力：
- 能力 1：具体内容、控制点、证据/假设
- 能力 2：具体内容、控制点、证据/假设

底层支撑：
- 数据 / 模型 / 系统 / 供应链 / 组织 / 治理 / 资本

战略结果：
- 用户价值
- 商业价值
- 风险约束
- 待确认假设
```

## 3. 选择架构视角

优先选择一个主视角。只有当单一视角无法解释对象时，才使用混合架构。

| 对象/任务 | 默认视角 | 适用原因 |
|---|---|---|
| 公司业务、平台商业模式 | 混合架构：生态 + 价值链 + 能力层 | 既要说明多方关系，也要说明赚钱、履约和反馈机制 |
| AI/Agent/技术平台 | 分层系统栈 | 入口、理解、编排、工具、模型、数据、治理有天然层级 |
| 产品体验、运营流程 | 流程架构 | 用户旅程、责任边界和反馈闭环最重要 |
| 行业格局、合作网络 | 生态地图 | 多主体价值交换、控制点和外部压力最重要 |
| 战略转型、组织能力 | 能力地图 | 重点是能力域、成熟度、支撑要素和约束 |
| 供应链、履约、交易系统 | 价值链/流程架构 | 输入、节点、库存、履约、成本和风险有明确流向 |

### 3.1 分层系统栈

用于平台、AI 系统、企业系统、技术架构。

- 顶部栏：用户、意图、输入通道、外部触发。
- 中间层：入口层、理解/规划层、能力/平台层、执行层。
- 底部层：模型、数据、基础设施、治理、安全、性能、成本。
- 右侧栏：价值、风险、战略影响、开放问题。

### 3.2 流程/价值链

用于业务流程、用户旅程、供应链、运营系统。

- 从左到右：触发 -> 接入 -> 理解/匹配 -> 决策/交易 -> 执行/履约 -> 反馈。
- 责任边界重要时使用泳道。
- 底部增加数据、治理、工具和可观测性支撑带。

### 3.3 生态地图

用于多边平台、行业格局、合作网络、开发者生态。

- 中心：平台或编排者。
- 内圈：核心能力和第一方服务。
- 中圈：合作伙伴、商家、开发者、渠道、供应链。
- 外圈：用户、监管、竞争者、资本、宏观力量。
- 箭头只表示价值交换、数据反馈、控制关系或商业化路径。

### 3.4 能力地图

用于战略、组织、转型、产品规划。

- 将能力归入 4-7 个领域。
- 每个领域必须有具体子能力、支撑资源和关键约束。
- 可用大小、位置或标签表达重要性、成熟度或优先级。

### 3.5 混合架构

用于公司业务、复杂平台、AI/Agent 生态。混合时必须有清晰主轴，不要把所有图形语法堆在一起。

推荐结构：

```text
左侧：角色/输入/市场力量
中间：主价值流或层级架构
底部：底层支撑和治理约束
右侧：战略影响、风险、决策点
```

## 4. 不同对象的默认建模方式

### 4.1 公司业务架构

重点不是把所有业务平均列出来，而是识别主飞轮、收入路径、履约/供应链能力、数据反馈和利润约束。

必须包含：

- 用户需求入口。
- 供给组织方式。
- 交易、匹配、履约或交付机制。
- 商业化路径：佣金、广告、订阅、商品销售、服务费、金融、硬件等。
- 底层能力：数据、技术、供应链、组织、品牌、资本。
- 关键风险：竞争、补贴、监管、成本、质量、安全、现金流。

好的公司业务图要回答：公司到底靠什么把需求、供给和履约组织到一起，哪里是控制点，哪里是利润压力。

### 4.2 产品/平台架构

重点是用户意图如何被捕获、理解、处理、执行和反馈。

必须包含：

- 用户角色和入口。
- 核心场景和任务。
- 产品模块和关键能力。
- 数据流、反馈流和运营干预点。
- 第三方生态或插件能力。
- 隐私、安全、可靠性、成本约束。

### 4.3 AI/Agent 系统架构

重点是意图如何进入系统，系统如何理解、规划、调用工具、执行动作，并在治理边界内反馈。

必须包含：

- 用户/应用入口。
- 上下文与记忆。
- 规划/编排层。
- 工具、插件、API、工作流。
- 模型、数据、检索、评估。
- 权限、审计、安全、成本、失败处理。

除非官方使用“Agent OS”等词，不要把自创强解释词当官方术语。

### 4.4 行业生态架构

重点是多主体如何交换价值、争夺控制点、受到外部力量约束。

必须包含：

- 关键参与方。
- 价值交换路径。
- 数据/资金/流量/供给控制点。
- 竞争者和替代方案。
- 监管、资本、技术周期等外部力量。
- 谁受益、谁被压缩、谁承担风险。

### 4.5 战略能力地图

重点是组织要具备哪些能力，能力之间如何支撑目标。

必须包含：

- 战略目标。
- 4-7 个能力域。
- 每个能力域的关键子能力。
- 支撑资源：人、流程、数据、系统、治理。
- 约束与优先级。
- 近期行动或决策点。

## 5. 正式绘图规范

### 5.1 默认画布

- 默认使用 16:9 一页画布，建议 1600x900 或等比尺寸。
- 输出 HTML/SVG 时必须能直接打开，不依赖远程运行时。
- 如果从 `C:\Windows\System32` 工作，产物保存到用户目录下的项目文件夹，不要写入系统目录。
- 文件名使用英文短横线，例如 `meituan-business-architecture.html`。

### 5.2 页面结构

正式图必须包含：

1. **标题**：主题 + 核心判断，不要只写对象名。
2. **副标题**：一句话说明这张图解释什么。
3. **事实/解读标签**：标明事实来源和分析边界。
4. **主轴**：阶段、层级、领域或角色。
5. **主画布**：模块、短标签、具体例子、方向箭头。
6. **侧栏**：价值、风险、战略影响、决策点或未知项。
7. **底部总结**：一句话总括 + 关键假设或来源边界。

### 5.3 模块规则

- 优先使用 5-8 个主层级或主领域。少于 4 个通常解释力不足，多于 9 个通常难读。
- 每个主模块包含标题和 2-4 个短要点。
- 每个模块都必须有具体内容。不要只写“平台层、数据层、能力层、生态层”。
- 同一层级的模块颗粒度要接近，不要把“用户”与“全球经济环境”放成同级主模块，除非图形语法明确区分。
- 如果一个模块需要超过 4 条要点，拆分或删掉低优先级信息。

### 5.4 箭头规则

箭头只表达真实方向，不作装饰。

允许表达：

- 输入进入系统。
- 价值流或业务流程。
- 能力调用。
- 数据反馈。
- 资金/商业化路径。
- 治理约束。

禁止：

- 为了显得复杂到处连线。
- 没有证据的强控制关系。
- 让箭头跨越太多模块造成视觉噪声。

### 5.5 配色语法

颜色用于编码含义，不用于装饰。

| 类别 | 推荐颜色 |
|---|---|
| 用户/输入 | 蓝色 |
| 理解/上下文 | 紫色 |
| 智能/规划/平台 | 绿色 |
| 模型/数据/基础设施 | 琥珀色或橙色 |
| 执行/履约/动作 | 黄色或金色 |
| 治理/安全/支撑体系 | 青绿色或石板灰 |
| 战略影响/风险 | 红色、玫红或强调色 |

避免整张图被单一色系主导。不要用大面积渐变、装饰圆球、抽象背景来替代结构表达。

### 5.6 文本密度

| 文本位置 | 建议长度 |
|---|---|
| 标题 | 8-18 个中文字符，必要时可更长但必须可读 |
| 副标题 | 20-45 个中文字符 |
| 层级标签 | 1-6 个字 |
| 卡片标题 | 2-8 个字 |
| 卡片要点 | 每条 4-14 个字 |
| 底部总结 | 一句话 |

文字必须在常见桌面和移动视口下不溢出、不重叠。必要时缩短文案，不要靠缩到不可读解决。

### 5.7 图形语法

- 圆角矩形：模块、服务、能力。
- 细边框卡片：支撑性子能力。
- 实线箭头：主流程或强依赖。
- 虚线箭头：反馈、可选关系或推断关系。
- 徽标：官方事实、观察事实、分析解读、待确认、风险。
- 侧栏卡片：所以怎样、价值、风险、开放问题。
- 底部深色条：总判断或运行原则。

## 6. 输出格式

### 6.1 默认正式交付

生成一份可直接打开的 HTML 文件，内嵌 CSS 和 SVG/HTML 图形，不依赖外部 JS。

交付时包括：

1. 文件链接。
2. 简短说明：基于公开事实、用户材料还是分析解读。
3. 显著改变架构的关键假设。
4. 如果用户要求发送飞书或其他渠道，完成真实发送后再报告结果。

### 6.2 快速文本草图

只有在用户明确要快速草图、或当前环境不能生成文件时，才使用 Mermaid 或文本结构。即便如此，也必须包含核心判断、主层级、流向和事实边界。

### 6.3 图片/PDF/PPT

用户要求图片、PDF 或 PPT 时，先生成 HTML/SVG 源文件，再导出或嵌入对应格式。导出后必须打开或渲染验证，不要只报告文件路径。

## 7. 反模式

出现以下情况必须返工：

- 图只是功能列表，没有系统逻辑。
- 标题是对象名，没有核心判断。
- 超过三分之一模块是“平台层、能力层、数据层、生态层”等空标签。
- 箭头没有真实方向，只是装饰。
- 事实和推测混在一起，可能误导读者。
- 把公司宣传词、自创术语包装成官方架构。
- 把公司业务图画成组织结构图，除非用户明确要组织架构。
- 视觉上依赖渐变、背景、装饰，而不是信息结构。
- 需要长篇解释才能看懂。
- 文件无法打开，或截图中有文字溢出、重叠、空白画布。
