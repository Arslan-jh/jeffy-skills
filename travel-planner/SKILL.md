# 旅行规划

## 默认策略

当用户只提供目的地时，直接按以下假设推进：
- 同时生成 3 天、5 天、7 天版本，以 5 天 4 晚作为主推荐。
- 节奏放松、不特种兵；每天通常 2-3 个核心锚点，留出餐饮、咖啡、酒吧和休息弹性。
- 默认中档舒适预算；用户指定穷游、奢华、亲子、商务、背包客、季节、同行人或特殊限制时按输入调整。
- 覆盖吃、喝、玩乐、购物、住、行、其他准备事项。
- 中文用户优先用中文写调研说明、HTML 界面文案和交付总结。
- 除非用户明确只要文字，否则创建单文件交互式 HTML。优先使用 `assets/interactive-itinerary-template.html`，复制到输出目录后替换其中的 `tripData`。
- 创建文件后，如飞书/lark-cli 可用，默认通过飞书发送一份；必须核对 `message_id`、上传结果或 Drive URL 后再声称已发送。

只有在目的地有歧义、用户需要固定日期预订、或缺失条件会实质影响方案时才追问。其他情况先声明假设并继续。

## 调研规则

推荐前必须检索当前公开信息。优先使用：
- 官方来源：旅游局、城市交通、机场/铁路、博物馆、公园、徒步线路、场馆官网。
- 地图和评价：Google Maps、Apple Maps、TripAdvisor、Booking、Agoda、Hotels.com、Expedia、OpenRice、Tabelog、Naver/Kakao、大众点评、小红书、Reddit、本地论坛等，按目的地和可访问性选择。
- 本地/编辑来源：Michelin、Eater、Time Out、本地报纸、本地餐饮作者、精品咖啡/鸡尾酒/精酿指南。
- 实用来源：汇率、政府旅行提醒、天气/气候、签证、安全、健康和通信信息。

营业时间、闭馆、价格、菜单、预约规则和联系方式以近期评价与官方页面为主。较早榜单只用于发现候选，必须再用当前来源复核。社交平台可用于判断氛围和发现候选，但不要把单条爆款内容当作事实依据。

不要编造营业时间、价格、预约规则、评分、坐标、电话、菜单、酒店房态或路线耗时。信息稀薄、季节性强、语言受限或评价两极分化时，明确标注不确定性。

## 行程设计

开头说明三件事：
1. “我先按以下假设规划”：天数、节奏、预算档位和旅行者类型。
2. “这条路线的逻辑是”：街区、路线和住宿基地如何组织。
3. “建议住在”：先推荐 1-3 个住宿区域，再列酒店或酒店类型。

分别设计 3/5/7 天，不要简单截短或填充：
- 3 天：一个最佳住宿基地、两个代表性街区、一个本地餐饮/市场簇、一顿强记忆点晚餐、一组咖啡/酒吧、一次 citywalk；徒步或室内攀岩只有在交通方便且不挤掉核心体验时加入。
- 5 天：主路线。平衡 citywalk、餐饮深度、一次户外/徒步、一次室内攀岩或本地活动、购物和休息时间。
- 7 天：增加慢街区、周边一日游、工作坊、更深入夜生活/酒吧、第二条徒步或风景路线、更多本地餐馆和咖啡厅；仍保持松弛，不把空档全部填满。

每日结构：
- 上午：一个主要街区、步行线、博物馆、市场或户外锚点。
- 下午：第二个锚点加咖啡/休息时间。
- 晚上：晚餐和附近酒吧/夜间活动。

交通上避免每天超过一次跨城/跨区大转移。抵达日和离开日默认放轻。已知日期时，把徒步和户外安排在天气更稳的一天。需要预订的餐厅、酒吧、活动和交通提前写进准备事项。

## 分类要求

吃：
- 写本地菜、市场小吃、面包/甜点/零食、随意餐、一顿更好的晚餐和预约提醒。
- 推荐点什么，而不是只写去哪。
- 可获得时提供菜单链接或官方/社交菜单来源。

喝：
- 包含精品咖啡、精酿、鸡尾酒、葡萄酒/清酒/茶/本地饮品。
- 说明氛围、预约需求、招牌饮品和酒精服务注意事项。

玩乐：
- 提供 citywalk 起止区域和地图停靠点。
- 室内攀岩馆必须核实现存、交通、日票/体验规则和位置便利性。
- 徒步线路写难度、交通、季节/天气风险和安全提示。
- 加入浴场、音乐现场、工作坊、运动、市场、节庆或街区仪式等本地活动。

购物：
- 以可食用、可使用或本地制造的伴手礼为主。
- 推荐实际购买点：市场、百货食品层、博物馆商店、独立店；机场只作为备选。

住：
- 先按便利度、交通、餐饮可达性、噪音、机场/车站衔接推荐区域。
- 再在当前信息充分时列代表性酒店或酒店类型。
- 优先干净、舒适、位置好、评价稳定、价格合理的住宿。
- 写清取舍：夜生活噪音、商务区晚上冷清、房间老、房间小、游客多或交通弱。

行：
- 说明抵离路线、市内交通票/支付方式、步行/打车、出租车、自行车/滑板车、必要时租车和城际交通。
- 标注末班车、现金支付、叫车限制等风险。

其他：
- 包含货币、汇率粗略核对、现金/刷卡、服务费/小费、插头、天气/打包、预订窗口、礼仪、安全、网络。
- 包含本地 App：地图、交通、打车、餐饮预订、翻译、支付、eSIM。
- 包含常用短句和发音：问候、谢谢、打扰、点餐、过敏/饮食禁忌、买单/支付、问路、紧急情况。

## HTML 交付要求

常规完整规划必须生成交互式 HTML，并告诉用户本地路径。页面必须能直接用浏览器打开；可通过 CDN 使用 Leaflet 地图，但如果地图瓦片加载失败，仍要保留文字路线和地点列表。

HTML 必须包含：
- 3/5/7 天切换。
- 分类筛选：吃、喝、玩乐、购物、住、行、准备。
- 每日选择器，切换后更新路线、停靠点和费用汇总。
- 动态人均预算计算器，支持人数、住宿档位和可选付费活动。
- 一张包含所有推荐地点的总地图，以及所选日期的路线连线或有序停靠列表。
- 地点卡片：地址、电话、官网、预订、菜单、地图、营业时间/来源、价格说明、推荐理由和注意事项。
- 来源面板，显示重要信息的来源和最后核对日期。
- 准备清单、本地 App、常用语。

坐标必须来自地图、地理编码或来源页核验。没有坐标时使用地图链接，不要编造精确点位。

费用模型：
- 住宿：每间每晚按人数分摊，提供经济/中档/舒适档。
- 吃喝：按日估算，并区分日常餐、更好晚餐、咖啡和酒吧。
- 活动：固定可选项，允许用户勾选。
- 市内交通：每日公共交通/打车估算。
- 城际或一日游交通：单独列项。
- 购物：作为可选预算，不默认计入总额，除非用户要求。

## 交付前验证

最终回复前必须检查：
- HTML 文件存在，且能渲染或至少内联脚本语法有效。
- 天数切换会改变天数和费用。
- 分类筛选会影响地点显示。
- 每个重要推荐都有来源链接，或明确写了不确定性。
- 如有地图瓦片联网、菜单缺失、季节性交通等限制，要在总结里说明。

回复用户时简短说明：文件路径、飞书发送结果或阻塞点、默认推荐方案及原因、重要预订/不确定性、已完成验证。
