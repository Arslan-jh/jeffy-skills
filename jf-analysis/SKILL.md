# 立体分析法深度研究

> **方法论定位**
> 立体分析法借鉴历时/共时分析、系统动力学因果链、竞争战略、案例研究与力量拆解分析。它不是把资料堆成百科，而是把一个对象放进“时间—力量—机制”的三维坐标里：先讲清楚路径，再讲清楚同一时刻的力场，最后讲清楚力场背后的内部逻辑。串联起来形成总体内容。

你正在执行一次立体分析法深度研究。默认交付物是：**一份完整 Markdown 报告 + 一份排版精美的 PDF 报告**。

## 0. 先判断任务是否需要本 Skill

触发后先做 10 秒判断：用户需要的是“系统研究”还是“简单回答”。

- **直接开始**：用户给出明确对象，并要求研究、分析、调研、竞品分析、摸清楚、搞懂、deep research。
- **最多追问一次**：对象不明确，或研究动机/关注点会显著影响结论。一次只问最关键的 1-3 个问题。
- **降级处理**：如果用户只是问“XX 是什么”，给简短解释即可，不要强行生成长报告。

若直接开始，记录以下研究设定：

| 字段 | 要求 |
|---|---|
| 研究对象 | 产品名、公司名、技术/概念名、人物名或事件名 |
| 对象类型 | 产品 / 公司 / 技术 / 概念 / 人物 / 事件 / 其他 |
| 研究动机 | 用户为什么现在关心它；没有就标注“未指定” |
| 特别关注点 | 用户指定的维度优先；没有就按通用三轴展开 |
| 时间基准 | 使用当前日期；涉及“最新/现在/最近”必须联网确认 |

## 1. 联网信息收集：先找事实，再做判断

这类报告的质量取决于信息密度、来源层级和证据链完整度。**必须联网获取信息**，不能只靠已有知识，也不能只靠几篇媒体报道或百科式资料撑起判断。

信息收集的目标不是“列来源”，而是为关键判断建立证据链：一手事实是什么，二手解释是什么，社区/用户声音如何补充，哪些材料互相冲突，哪些地方只能推测。

### 1.1 并行采集分工

如果环境支持子 Agent，可并行分工；不支持时按同样结构顺序执行。

| 分工 | 目标 | 典型线索 |
|---|---|---|
| X轴：时间线 | 起源、发展阶段、关键事件、因果链 | 首次发布/成立、融资、版本、组织变动、战略转向、危机、并购、政策冲击 |
| Y轴：力场 | 关键时间截面上同时存在的外部/内部力量 | 竞品、资本、技术、市场、政策、人物、舆论、供应链、宏观周期、地缘政治 |
| Z轴：机制 | 对关键力量做内部拆解 | 创始人访谈、内部信/公开信、财报、监管文件、GitHub Issues、社区讨论、论文、行业深访 |

### 1.2 子 Agent 联网提示词

给每个子 Agent 的任务里加入以下指引：

> 你需要联网获取信息。先用 WebSearch 发现来源和关键词，再用 WebFetch 读取具体页面。多组关键词、多轮搜索，不要只搜一次。每个关键判断至少尝试找到一手来源、可靠二手来源和反向证据。不要只收集支持结论的材料，也要寻找失败案例、负面评论、监管/诉讼/社区争议、竞品反驳和历史误判。涉及学术概念、算法、AI 模型、技术范式时必须查 arXiv，可用 `https://export.arxiv.org/api/query?search_query=all:关键词&max_results=10`。输出时区分事实、引用、解释和推测，并标注证据强弱。

### 1.3 来源优先级

| 信息类型 | 优先来源 |
|---|---|
| 产品更新/技术决策 | 官方博客、Release Notes、GitHub commits/issues、创始人公开发言 |
| 融资/商业数据 | 公司公告、SEC/交易所文件、工商信息、投资方公告、可信数据库 |
| 用户口碑 | GitHub Issues、Reddit、Hacker News、X/Twitter、知乎、应用商店评价、论坛长帖 |
| 行业格局 | 权威媒体原创报道、分析师报告、财报、行业协会数据 |
| 学术/技术原理 | arXiv、会议论文、官方技术报告、标准文档 |
| 力量内部机制 | 访谈、内部信、组织历史、诉讼/监管材料、关键人物履历、长期社区讨论 |

### 1.4 检索质量门槛

正式写作前必须先输出“检索问题清单”和“证据链矩阵”。检索问题清单用于约束搜索方向，避免看到什么写什么；证据链矩阵用于确认每个核心判断是否已经有足够材料支撑。

每个核心判断至少需要两类不同来源支撑；因果判断必须同时包含事实节点、行为证据和外部解释。缺少任一类材料时，不要写成确定结论，必须降低置信度或标注资料缺口。

每个研究对象必须主动检索反向材料，包括失败案例、负面评价、监管/诉讼、用户流失、竞品反驳。找不到反向材料时，也要说明检索过哪些方向，不能默认不存在。

所有来源按 A-E 分级：A 类是一手材料，B 类是行为证据，C 类是可靠二手，D 类是社区/用户声音，E 类是弱来源。A/B/C 可支撑正式判断，D 用于补充用户声音，E 只能作为线索，不能单独支撑结论。

涉及“最新/现在/最近”时，必须记录来源发布日期和访问日期。无法确认发布时间或访问状态的材料，只能作为辅助线索，不能支撑最新状态判断。

如果证据链不足，停止正式写作，改为输出资料缺口清单。不要为了完成报告而用单一来源、媒体转述或未经验证的社区说法补足关键判断。

### 1.5 证据链要求

写作前先为核心判断建立“证据链矩阵”。每个重要判断尽量同时覆盖以下几类材料：

| 证据层级 | 作用 | 示例 |
|---|---|---|
| 一手事实 | 确认发生了什么 | 官方公告、产品文档、财报、监管文件、论文、GitHub commit/issue、法庭文件 |
| 行为证据 | 观察参与方真实选择 | 定价变化、组织调整、招聘方向、路线图变动、版本发布、渠道变化 |
| 外部解释 | 理解行业语境 | 权威媒体原创报道、分析师报告、行业数据库、专家访谈 |
| 用户/社区声音 | 识别真实摩擦 | Reddit、Hacker News、GitHub Issues、应用商店评价、论坛长帖、客户案例 |
| 反向材料 | 防止单边叙事 | 失败案例、负面评价、竞品反驳、监管处罚、诉讼、流失用户反馈 |
| 缺口标注 | 明确边界 | 找不到可靠来源、口径不一致、时间线冲突、样本偏差 |

核心判断至少要有两类不同来源支撑；关键结论最好有一手来源或行为证据支撑。只有媒体转述、二手总结或单一社区帖子时，不要写成确定结论。

**硬规则：没有证据链矩阵，不进入正式写作。**

如果暂时无法建立证据链矩阵，只能继续补搜，或向用户说明“当前信息不足以支撑深度分析”，并交付资料缺口清单。不要在证据链不足时直接写完整报告。

最低证据门槛：

1. **事实性节点**：至少找到一手来源或可靠二手来源；找不到时标注“暂未找到一手来源”。
2. **因果判断**：至少同时有事实节点、行为证据和外部解释支撑；缺任一项时降低置信度。
3. **机制判断**：至少要有行为证据或当事方材料支撑，不能只靠媒体解读。
4. **用户/市场判断**：至少结合用户/社区声音与可观察指标，不能只引用个别评论。
5. **争议性判断**：必须列出反向材料或替代解释；找不到反向材料也要说明检索过哪些方向。
6. **最新状态**：涉及“现在/最新/最近”时，必须用当前日期下可访问的来源确认。

### 1.6 信息充分性自检

进入写作前检查：

- X轴：能否讲出从起源到今天的完整因果链？关键节点之间的“因为→所以”是否成立？
- Y轴：是否只选择真正改变路径的关键截面？每个截面是否能讲清主导力量、制约力量和被忽略变量？
- Z轴：是否只拆最关键的少数机制？是否能解释“为什么会这样”，而不是把力量继续拆成碎片？
- 来源：每个关键判断是否有证据链支撑？有没有一手来源、行为证据或反向材料？
- 边界：信息不足、口径冲突、样本偏差是否明确标注？推测是否说明依据和置信度？

信息不够就继续补搜；不要为了开始写作而凑合。

### 1.7 视觉表达规划

正式写作前必须先做“视觉表达规划”。`jf-analysis` 不是纯文字报告；当图形能明显降低理解成本、揭示关系或压缩复杂信息时，应主动使用图表、概念图或结构图。图表数量不固定，由内容复杂度、证据密度和读者理解成本决定。

视觉表达遵循三条原则：

1. **图表服务判断**：每张图必须回答一个分析问题，例如路径如何演化、力量如何交织、机制如何运转、变量如何影响未来。不能为了美观、填充版面或显得高级而画图。
2. **有证据才量化**：只有当数据口径可靠、可比较、能支撑结论时才做柱状图、折线图、气泡图等量化图。数据不足时，用机制图、力场图、时间轴、二维矩阵或概念图，不要编造数值。
3. **文字和图互相补充**：图表之后必须用短段解释“这张图说明什么、哪里最关键、最大不确定性是什么”。不要让图表替代因果分析。

推荐先做一个图表预算：

| 报告复杂度 | 建议图表数量 | 典型图表 |
|---|---:|---|
| 简短对象 / 单一产品 | 2-3 张 | 时间轴、机制链、三轴概念图 |
| 标准公司 / 产品 / 行业分析 | 4-6 张 | 时间轴、力场图、对比矩阵、变量表、三轴概念图 |
| 复杂行业 / 长历史 / 多方博弈 | 6-8 张 | 阶段演进图、多主体力场图、因果回路、战略坐标、证据热力图、三轴概念图 |

图表类型选择：

| 分析任务 | 优先图形 | 使用条件 |
|---|---|---|
| 说明路径如何演化 | 时间轴、阶段演进图、S 曲线 | X轴节点有清晰阶段或拐点 |
| 说明多方力量如何作用 | 力场图、利益相关方网络、四象限矩阵 | Y轴存在多主体博弈、共振或牵制 |
| 说明机制如何运转 | 因果链、反馈回路、飞轮/反飞轮、漏斗 | Z轴需要解释内部动力和约束 |
| 说明竞争位置 | 二维坐标、雷达图、能力矩阵 | 有明确比较维度，且各对象可比较 |
| 说明证据强弱 | 证据矩阵、置信度热力图 | 核心判断存在不同来源等级 |
| 说明未来变量 | 驱动变量地图、情景触发图、领先指标仪表盘 | 未来判断依赖少数关键变量组合 |
| 汇总三轴洞察 | X/Y/Z 三轴概念图、三层解释模型 | 报告完成 X/Y/Z 后必须至少考虑一次 |

正文中的图表优先使用 `figure` + inline SVG，确保 Markdown、HTML 和 PDF 都能渲染。不要依赖 Mermaid 运行时；如使用 Mermaid，必须先渲染为 SVG 后嵌入。图表代码应尽量短，避免复杂装饰；颜色保持红灰为主，辅助色只用于必要分类。

图表示例骨架：

```html

  <figcaption>图 1：一句话说明图表回答的问题</figcaption>

    <text x="130" y="122" text-anchor="middle" font-size="18" font-weight="700">阶段一</text>
    <path d="M230 115 L330 115" stroke="#c00000" stroke-width="2"/>
  </svg>
</figure>
```

## 2. X轴：时间线因果链

X轴回答：**它是怎样一步步走到今天的？**

### 2.1 写什么

- **起源追溯**：时代背景、行业状态、技术条件、需求缺口、创始团队/提出者背景、直接触发因素。
- **诞生节点**：首次发布/成立/提出的明确时间、地点/环境、早期形态、初始定位。
- **关键节点**：版本、融资、组织变化、战略转向、外部冲击、危机、监管、并购、用户增长拐点。
- **因果链**：每个节点都要回答“发生了什么 → 为什么重要 → 它如何导向下一步”。
- **阶段划分**：按自然拐点分阶段，每个阶段写核心驱动力、核心矛盾、关键人物/决策和阶段结果。

### 2.2 怎么写

不要写成大事记。好的 X 轴像一条故事线：每个节点都带着前一个节点的遗留问题，同时埋下下一个节点的条件。

推荐节点结构：

```text
时间 / 节点名
- 发生了什么：用事实说清楚事件
- 为什么发生：直接原因与背景约束
- 为什么重要：它改变了资源、认知、路径或竞争位置中的哪一项
- 导向机制：它如何把对象推向下一个阶段
- 证据来源：链接或材料名
```

X轴通常适合配一张图。优先选择时间轴、阶段演进图、关键转折链或 S 曲线。图中不要塞满所有事件，只保留真正改变路径的 4-7 个节点；普通年表、背景事实和次要版本变化放入正文或附录。图后用一段话解释最大转折点和它如何改变后续路径。

### 2.3 篇幅

建议 4,000-12,000 字。历史长、节点多的对象靠近上限；新生对象可以较短，但不能跳过关键因果。

## 3. Y轴：关键节点力场

Y轴回答：**在每个关键时间截面，哪些力量同时在作用，它们如何交织？**

### 3.1 先选关键截面

从 X轴中挑 2-4 个最能改变走向的节点作为 Y轴截面。不要每一年都展开，也不要把所有阶段平均铺开。宁愿少选，但每个截面都要能回答：如果没有这个截面，整个对象的路径解释是否会明显变弱？

优先选择这几类截面：

1. **方向改变点**：战略、产品路线、技术路线或商业模式发生明显转向。
2. **外部压力点**：监管、资本周期、平台政策、竞品冲击、宏观环境显著改变选择空间。
3. **资源重组点**：融资、并购、组织调整、关键人物变动、生态关系变化。
4. **认知反转点**：市场叙事、用户口碑、开发者态度或行业评价发生转折。

### 3.2 力量类型

根据对象选择，不必强行覆盖全部。Y轴的重点不是“分类完整”，而是讲清关键时刻的主导力量组合。每个截面通常只保留 3-5 个真正改变局面的力量。

| 力量 | 关注问题 |
|---|---|
| 竞品 | 同期对手是谁，采取什么路线，如何施压或被迫响应 |
| 资本 | 资金、估值、投资方诉求、现金流、控制权如何影响选择 |
| 技术 | 底层技术成熟度、路线选择、人才供给、基础设施、技术债 |
| 市场 | 用户需求、渠道、定价、留存、渗透率、购买/采用阻力 |
| 政策/监管 | 准入、牌照、数据/隐私、安全、诉讼、补贴或限制 |
| 人物/组织 | 创始人、核心团队、组织文化、人事变动、决策风格 |
| 舆论/社区 | 媒体叙事、用户口碑、开发者社区、意见领袖、危机传播 |
| 供应链/生态 | 上下游、平台依赖、合作伙伴、分发渠道、生态锁定 |
| 宏观/地缘/文化 | 利率周期、消费能力、国际关系、社会价值观、人口结构 |

### 3.3 写法要求

Y轴不是“力量清单”，而是“力场叙事”。每个截面要讲出一个“主矛盾”，再围绕主矛盾展开力量交织。不要把竞品、资本、技术、市场、政策逐项平铺。

- 谁在主动塑造格局，谁在被动响应？
- 哪些力量互相强化，哪些力量互相抵消？
- 哪个力量看似边缘，却改变了后来的路径？
- 如果只看研究对象本身，会漏掉什么外部压力？
- 当时真正的约束是什么？谁的选择空间最大，谁其实没有选择？

推荐截面结构：

```text
关键截面：时间 / 事件
- 当时的局面：一句话说明这个截面为什么关键
- 主矛盾：这个截面最核心的张力是什么
- 主要力量：3-5 个真正起作用的力量
- 交织关系：这些力量如何博弈、共振或互相牵制
- 主动权判断：谁在推动，谁在响应，谁误判了
- 被忽略变量：只看研究对象本身会漏掉什么
- 对 X轴的影响：这个力场如何改变下一段因果链
```

Y轴通常适合配力场图或关系图。可用多主体网络图、四象限矩阵、压力—响应图、利益相关方图，呈现“谁推动、谁制约、谁被迫响应”。图中只放关键力量和主要作用方向，不要把所有分类都画进去。若力量之间没有明确方向或互动证据，不要强行画关系线。

### 3.4 篇幅

建议 3,000-8,000 字。复杂对象可多写；但宁愿少选截面，也要把每个截面的主矛盾、力量交织和路径影响讲透。

## 4. Z轴：力量内部机制拆解

Z轴回答：**关键力量为什么这样行动？内部是什么结构？受到什么约束？如何与其他力量相互影响？**

### 4.1 拆解对象

只拆最重要的少数机制。每个关键截面最多选择 1-2 个力量做 Z轴深拆，整篇报告通常只拆 3-5 个核心机制。不要把所有力量都拆一遍，也不要把一个机制继续拆成大量碎片。

选择 Z轴对象时，优先问三个问题：

1. 它是否解释了 X轴中最关键的路径选择？
2. 它是否解释了 Y轴中最难理解的力量互动？
3. 如果不拆它，报告是否只能停留在表面叙事？

可拆对象示例：

- 竞品：为什么它选择这条战略，而不是另一条？它的组织能力和既有业务如何限制它？
- 资本：投资方要的是什么？融资环境、估值压力、退出路径如何塑造决策？
- 技术：这条技术路线为什么当时看起来合理？技术债、人才结构、基础设施如何锁定路径？
- 市场：真实需求是什么？用户为什么买/不用/流失？渠道和定价如何改变采用曲线？
- 政策：监管者的目标、约束和窗口期是什么？为什么在那个时间点发生？
- 人物/组织：关键人物的经历、利益、性格、权力位置如何影响选择？组织惯性如何放大个人判断？
- 舆论/社区：叙事如何形成、扩散、反噬？核心用户和大众认知是否分裂？

### 4.2 每个机制必须回答

1. **机制名称**：用一句话命名这个机制，例如“平台依赖导致的路线锁定”“资本退出压力压缩战略耐心”。
2. **内部结构**：这股力量内部有哪些玩家、派别、资源或约束？它不是铁板一块。
3. **行为动机**：它为什么这样行动？短期收益和长期目标分别是什么？
4. **约束条件**：什么限制了它的选择空间？技术、资金、组织、法律、路径依赖还是声誉？
5. **互动关系**：它如何影响其他力量，又如何被其他力量改变？
6. **证据等级**：基于一手材料、可靠报道、社区证据，还是推测？推测必须明说。
7. **解释增量**：拆完这个机制后，读者比只看表面事实多理解了什么？

### 4.3 避免两种错误

- **资料堆砌**：不要把人物履历、融资表、参数表直接塞进来；要解释“为什么这些事实改变了行为”。
- **空洞心理分析**：不要凭空揣测动机。没有证据就写“该信息暂缺”或“推测，依据是……”。
- **碎片化拆解**：不要把每个力量都拆成“动机/约束/互动”的小表格。Z轴要服务于解释，不是为了形式完整。

Z轴适合配机制图。优先用因果链、反馈回路、飞轮、反飞轮、漏斗或约束结构图，说明内部机制如何把动机、资源、约束和外部压力连接起来。机制图必须有方向感，至少回答“输入是什么、约束是什么、输出是什么、反馈如何发生”。没有明确机制闭环时，用分层结构图，不要伪装成飞轮。

### 4.4 篇幅

建议 2,000-5,000 字。Z轴贵在深，不在全。

## 5. 三轴交汇洞察

这是报告的精华。交汇洞察不是前文摘要，而是基于三轴材料产出的新判断。

必须回答：

1. **历史如何塑造今天**：X轴哪些路径选择决定了当下位置？
2. **力场如何改写历史**：Y轴哪些外部/内部力量加速、延缓或扭转了因果链？
3. **机制揭示了什么**：Z轴让我们看到哪些表面叙事解释不了的东西？
4. **优势和劣势的根源**：今天的强项/短板分别能追溯到哪些早期选择或约束？
5. **未来主导走向**：先识别关键驱动变量，再收敛成一个最有解释力的未来主线。不要拆成多个并列方向，也不要使用情绪标签。

三轴交汇处必须考虑生成一张“X/Y/Z 三轴概念图”。这张图不是方法论示意图，而是基于本次研究内容提炼出的解释模型：X轴放关键路径或阶段，Y轴放关键力场或主导力量，Z轴放底层机制或约束，三者汇入一个核心判断或未来主线。图形可以是三层漏斗、三轴坐标、中心辐射图、因果汇流图或“路径—力场—机制”三栏模型，具体形式由内容决定。

三轴概念图必须满足：

1. 图中每个节点都来自前文证据链或明确推测，不能新增未经解释的概念。
2. 中心判断必须是一句话机制命名，例如“合规成本换取存量锁定”“流量入口被交易效率重新定价”。
3. 图后必须用 1-2 段解释：X/Y/Z 如何共同导向该判断，以及最可能错在哪个前提。
4. 如果研究对象过于简单，无法形成有信息增量的三轴概念图，可以省略，但必须在附录或交付说明中说明原因。

每条洞察建议写成：

```text
洞察：一句话判断
依据：来自 X/Y/Z 哪些事实
置信度：高/中/低，以及为什么
可能错在哪里：如果判断失效，最可能是哪个前提错了
```

### 5.1 未来主导走向写法

未来主导走向不能拍脑袋，也不能只写“增长/停滞/衰退”这种低信息量判断。先做驱动变量拆解，再收敛成一个最有解释力、最值得跟踪的未来主线。

步骤：

1. **列出关键驱动变量**：影响未来走向的 3-6 个变量，例如监管强度、技术突破速度、资金成本、用户迁移成本、平台政策、组织执行力、关键人物变化。
2. **区分确定性与不确定性**：哪些趋势已经比较确定，哪些变量仍然高度不确定。
3. **筛出主导变量组合**：找出最能解释未来路径的 1-2 个变量组合，而不是搭出多个并列方向。
4. **形成一个主导走向**：用具体机制命名，说明为什么这条主线比其他可能性更值得关注。
5. **写触发条件和领先指标**：说明什么事件会强化这条主线，哪些早期信号说明它正在发生。
6. **写反证条件**：什么证据出现时，这条主线应该被修正或推翻。
7. **写替代解释**：简短说明还有哪些次要可能性，但不要展开成并列主线。

推荐结构：

```text
主导走向：用机制命名，例如“合规成本换取存量锁定”
- 核心逻辑：关键变量如何组合，为什么这条主线最值得关注
- 触发条件：哪些事件或数据变化会强化这条主线
- 领先指标：未来 3-12 个月应该观察哪些信号
- 受益方/受损方：哪些力量会被强化，哪些力量会被削弱
- 反证条件：什么情况出现说明这条主线需要修正或推翻
- 替代解释：还有哪些次要可能性，为什么暂不作为主线
- 置信度：高/中/低，以及最大不确定性
```

低质量写法示例：未来可能继续增长、也可能遇到监管风险、也可能被竞品超越。

高质量写法示例：如果监管口径收紧但用户迁移成本仍高，对象的主导走向可能是“合规成本换取存量锁定”；早期信号是合规团队扩张、产品功能收缩、价格体系向企业客户倾斜、开发者社区增长放缓。替代解释可以简短提到“开放生态反攻”或“低价竞品迁移”，但不展开成并列主线。

篇幅建议 1,500-3,000 字。

## 6. 写作风格

目标是“严谨但有人味”的研究报告：结构清楚、事实扎实、判断锋利、读起来有叙事张力。

### 6.1 应该这样写

- **叙事驱动**：X轴有故事弧线，不是流水账。
- **具体胜过概括**：用数字、时间、人物、产品细节和真实用户声音支撑判断。
- **先事实后判断**：观点可以鲜明，但必须有证据；推测要标注。
- **节奏有变化**：长短句交替，关键判断可以单独成段。
- **自然解释背景**：背景知识要嵌在叙事里，不要像百科词条。
- **回环呼应**：开头提出的关键矛盾，结尾要回到它。

### 6.2 绝对禁区

- 套话：首先、其次、最后、综上所述、值得注意的是、不难发现。
- 空洞词：赋能、抓手、闭环、生态化布局、降本增效但无细节。
- 教科书开头：在当今快速发展的时代、随着技术不断进步。
- 高频 AI 腔：本质上、换句话说、不可否认、这意味着什么。
- 空泛主体：不要写“某 AI 工具”“某模型”，要写具体名称。
- 编造：搜不到就标注暂缺；冲突信息就并列说明。

## 7. 报告结构与交付

### 7.1 硬性排版规范

生成正式报告时必须遵守以下结构与排版规则：

1. **正文采用“短正文 + 附录”结构**。研究设定、资料边界、证据矩阵和方法论不放在正文前部，统一放到附录，避免打断阅读。
2. **正文只保留 6 个一级章节**：核心结论、时间线因果链、关键力场、底层机制、未来走势、结论。正文不使用三级及以下标题。
3. **正文最多使用两级标题**：`##` 和 `###`。除非用户明确要求超长学术报告，不使用 `####` 和 `#####`。
4. **一级章节使用中文序号**，格式为 `## 一、核心结论`、`## 二、为什么会走到今天：时间线因果链`。不要使用无序号标题。
5. **二级标题编号必须与一级章节对应**：`## 一、...` 下只能使用 `### 1.1`、`### 1.2`；`## 二、...` 下只能使用 `### 2.1`、`### 2.2`；以此类推。不要出现“二、下面写 3.1”这类编号错位。附录使用 `### A1`、`### A2`、`### A3`，不参与正文数字编号。
6. **不要把分析动作标题化**。例如“当时局面、主矛盾、交织关系、约束条件、解释增量、置信度”都应写进自然段；只有分析对象、阶段或机制发生变化时才开新标题。
7. **段落首行缩进 2 个中文字符**。Markdown 正文段落不要手动添加全角空格；HTML/PDF 样式中统一用 CSS `p { text-indent: 2em; }` 实现。标题、表格、代码块、引用块和来源链接列表不缩进。
8. **尽量不用项目符号**，尤其避免使用 `·`。能写成自然段就写成自然段；必须列举时，优先使用表格或编号列表。列表不可替代分析，每一项都要包含判断或因果解释。
9. **图表嵌入正文**。图表应放在它解释的段落附近，而不是集中堆到附录。每张图必须有编号、标题/图注和解释段。图表不得替代证据来源；来源仍放在正文括注或附录资料来源中。
10. **PDF 友好**。标题层级不要跳级；长 URL 统一放到附录“资料来源”章节；正文中用简短来源名或脚注式括号，不要让 URL 打断段落阅读。图表优先使用 inline SVG 或已渲染图片，避免依赖运行时脚本。

| 事件 | 前因、爆发、扩散、后果 | 各参与方、媒体、监管、市场反应 | 参与方诉求、信息不对称、激励结构 |
